## 核心知识一：Linux命令行数据处理

### 常用管道命令

S515课程强调==Shell与Jupyter的集成==，通过 `%%bash` 魔法命令在Notebook中执行Shell操作：

| 命令 | 用途 | 示例 |
|------|------|------|
| `head -n` / `tail -n` | 查看文件首/尾n行 | `head -3 data.csv` |
| `sed -n 'a,bp'` | 提取第a到b行 | `sed -n 15,20p data.csv` |
| `grep` | 模式匹配筛选行 | `grep 'Data,9' file.csv \| wc -l` |
| `awk -F,` | 按分隔符提取字段 | `awk -F, '{print $5,$8}' file.csv` |
| `wc -l` | 统计行数 | 管道末端统计匹配行数 |

> [!example] 骑行数据清洗 (S515 HW1)
> 从非标准CSV中提取结构化数据：原始文件每行包含 `Type,Local Number,Message` 前缀 + 键值对三元组。用 `awk` 按位置提取 timestamp/lat/long/distance/altitude/speed 六个字段，重构为标准CSV。

### 三种实现对比

```python
# 方法1: Linux awk (最快)
awk -F, '{if($2==9){print $5,$8,$11,$14,$17,$20}}' ride.csv

# 方法2: Python csv模块
with open('ride.csv') as f:
    for row in csv.reader(f):
        if row[0]=='Data' and row[1]=='9': ...

# 方法3: Pandas (最简洁)
df = df[(df['Type']=='Data') & (df['Local Number']==9)]
df_sorted = df[['Value 1','Value 2','Value 3','Value 4','Value 5','Value 6']]
```

> [!tip] 性能排序
> 对于简单的行筛选+列提取：Shell命令 > Python原生 > Pandas。但复杂分析场景下Pandas的表达力远超前两者。

## 核心知识三：MapReduce与mrjob

### MapReduce编程模型

==MapReduce==将大规模数据处理分为三个阶段：

1. **Map**：将输入拆分为键值对 $(k_1, v_1) \rightarrow [(k_2, v_2)]$
2. **Shuffle**：按键分组，相同键的值聚合到一起
3. **Reduce**：对每个键的值列表进行聚合 $(k_2, [v_2]) \rightarrow (k_3, v_3)$

### mrjob实现模式

```python
from mrjob.job import MRJob
from mrjob.step import MRStep

class MRWordCount(MRJob):
    def mapper(self, _, line):
        yield "chars", len(line)
        yield "words", len(line.split())
        yield "lines", 1
    def reducer(self, key, values):
        yield key, sum(values)
```

### 多步骤MapReduce

> [!example] 最频繁单词检索 (S516 HW1)
> 两步MR流水线：Step1的mapper提取单词并yield `(word, 1)`，reducer求和得 `(word, count)`；Step2的reducer用 `max()` 找最高频词。莎士比亚全集最频繁词："thou" (5479次)。

```python
class MRMostUsedWord(MRJob):
    def steps(self):
        return [
            MRStep(mapper=self.mapper_get_words,
                   reducer=self.reducer_count_words),
            MRStep(reducer=self.reducer_find_max_word)
        ]
```

> [!tip] Combiner的作用
> Combiner在mapper端做局部聚合，理论上减少网络传输。但S516实验发现对小数据集，==去掉combiner反而更快==（从7.3s降至5.7s），因为combiner增加了额外的函数调用开销。

## 核心知识四：Spark/PySpark

### RDD vs DataFrame

| 特性 | RDD | DataFrame |
|------|-----|-----------|
| 抽象层次 | 低级，函数式 | 高级，声明式 |
| 优化 | 无自动优化 | Catalyst优化器 |
| Schema | 无 | 有，支持SQL |
| 适用场景 | 非结构化数据 | 结构化/半结构化 |

### Spark核心操作

| 类型 | 操作 | 描述 |
|------|------|------|
| Transformation (惰性) | `map`, `filter`, `flatMap` | 定义计算图，不立即执行 |
| Transformation (惰性) | `reduceByKey`, `groupByKey` | 按键聚合 |
| Action (触发执行) | `collect`, `count`, `take` | 触发计算并返回结果 |

> [!tip] 惰性求值
> Spark的Transformation不立即计算，仅在Action触发时才执行。这允许Spark优化整个计算图（DAG），避免不必要的中间结果。

## 核心知识五：SQL规范化与数据库设计

### 范式与规范化

| 范式 | 要求 | 目标 |
|------|------|------|
| 1NF | 每列原子值，无重复组 | 消除重复列 |
| 2NF | 满足1NF + 非键属性完全依赖主键 | 消除部分依赖 |
| 3NF | 满足2NF + 无传递依赖 | 消除传递依赖 |

### PostgreSQL查询优化

> [!example] 规范化表查询 (S611 Assignment)
> 非规范化表 `home_value_by_zip`（446万行）vs 规范化多表JOIN查询。要求用JOIN重写聚合查询，且执行时间 < 4秒。关键是利用索引和正确的JOIN顺序。

```sql
SELECT state, avg(ym_val) AS average
FROM home_value_by_zip
GROUP BY state ORDER BY average DESC LIMIT 10;
-- 规范化版需将此改为多表JOIN实现
```

### 数据库设计要点

- ==反规范化==适用于读多写少的分析场景
- 索引策略：在 `WHERE`、`JOIN`、`GROUP BY` 列上建索引
- 大表查询优先用 `LIMIT` 限制结果集

## 方法选择指南

```
数据规模判断：
  ├── < 1GB → Pandas (单机内存)
  ├── 1-100GB → PySpark DataFrame
  └── > 100GB → Spark集群 + 分布式存储

存储选择：
  ├── 事务型 (OLTP) → PostgreSQL (规范化设计)
  ├── 分析型 (OLAP) → 列式存储 / 反规范化
  └── 半结构化 → JSON/MongoDB

处理模式：
  ├── 批处理 → MapReduce / Spark Batch
  └── 流处理 → Spark Streaming
```

## 实战洞察

- Linux管道处理是数据工程师的基本功，对于TB级日志文件的预处理比Pandas快一个数量级
- `%%bash` 魔法命令仅支持非交互式命令，`nano`/`vi` 等编辑器无法使用
- mrjob适合学习MapReduce概念，但生产环境直接用Spark
- Pandas的 `groupby` + `first()` 是时间序列降采样的高效模式
- 规范化数据库设计的核心权衡：数据一致性 vs 查询性能
- 透视表操作前务必确认聚合函数（`mean`/`sum`/`count`），错误的聚合会导致误导性结果
- 大数据处理的第一步永远是==采样验证==，在小数据集上调试后再提交集群作业
